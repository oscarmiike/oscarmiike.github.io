Good pull requests -\/ patches, improvements, new features -\/ are a fantastic help. They should remain focused in scope and avoid containing unrelated commits.

{\bfseries{Please ask first}} before embarking on any significant pull request (e.\+g. implementing features, refactoring code), otherwise you risk spending a lot of time working on something that the project\textquotesingle{}s developers might not want to merge into the project.

Please adhere to the coding conventions used throughout a project (indentation, accurate comments, etc.) and any other requirements (such as test coverage).

Adhering to the following this process is the best way to get your work included in the project\+:


\begin{DoxyEnumerate}
\item \href{http://help.github.com/fork-a-repo/}{\texttt{ Fork}} the project, clone your fork, and configure the remotes\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\#\ Clone\ your\ fork\ of\ the\ repo\ into\ the\ current\ directory}
\DoxyCodeLine{git\ clone\ https://github.com/<your-\/username>/bower}
\DoxyCodeLine{\#\ Navigate\ to\ the\ newly\ cloned\ directory}
\DoxyCodeLine{cd\ bower}
\DoxyCodeLine{\#\ Assign\ the\ original\ repo\ to\ a\ remote\ called\ "{}upstream"{}}
\DoxyCodeLine{git\ remote\ add\ upstream\ https://github.com/bower/bower}

\end{DoxyCode}

\item If you cloned a while ago, get the latest changes from upstream\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ checkout\ master}
\DoxyCodeLine{git\ pull\ upstream\ master}

\end{DoxyCode}

\item Create a new topic branch (off the main project development branch) to contain your feature, change, or fix\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ checkout\ -\/b\ <topic-\/branch-\/name>}

\end{DoxyCode}

\item Make sure to update, or add to the tests when appropriate. Patches and features will not be accepted without tests. Run {\ttfamily npm test} to check that all tests pass after you\textquotesingle{}ve made changes.
\item Commit your changes in logical chunks. Please adhere to these \href{http://tbaggery.com/2008/04/19/a-note-about-git-commit-messages.html}{\texttt{ git commit message guidelines}} or your code is unlikely be merged into the main project. Use Git\textquotesingle{}s \href{https://help.github.com/articles/interactive-rebase}{\texttt{ interactive rebase}} feature to tidy up your commits before making them public.
\item Locally merge (or rebase) the upstream development branch into your topic branch\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ pull\ [-\/-\/rebase]\ upstream\ master}

\end{DoxyCode}

\item Push your topic branch up to your fork\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{git\ push\ origin\ <topic-\/branch-\/name>}

\end{DoxyCode}

\item \href{https://help.github.com/articles/using-pull-requests/}{\texttt{ Open a Pull Request}} with a clear title and description.
\item If you are asked to amend your changes before they can be merged in, please use {\ttfamily git commit -\/-\/amend} (or rebasing for multi-\/commit Pull Requests) and force push to your remote feature branch. You may also be asked to squash commits.
\item If you are asked to squash your commits, then please use {\ttfamily git rebase -\/i master}. It will ask you to pick your commits -\/ pick the major commits and squash the rest.
\end{DoxyEnumerate}

{\bfseries{IMPORTANT}}\+: By submitting a patch, you agree to license your work under the same license as that used by the project.

If you have commit access, please follow this process for merging patches and cutting new releases.\hypertarget{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_internal_a_p_i_2_internal_a_p_i_2wwwroot_2lib_2f649a80c2bb62ca06354bea1a086dc0e_autotoc_md699}{}\doxysubsection{\texorpdfstring{Reviewing changes}{Reviewing changes}}\label{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_internal_a_p_i_2_internal_a_p_i_2wwwroot_2lib_2f649a80c2bb62ca06354bea1a086dc0e_autotoc_md699}

\begin{DoxyEnumerate}
\item Check that a change is within the scope and philosophy of the project.
\item Check that a change has any necessary tests and a proper, descriptive commit message.
\item Checkout the change and test it locally.
\item If the change is good, and authored by someone who cannot commit to {\ttfamily master}, please try to avoid using Git\+Hub\textquotesingle{}s merge button. Apply the change to {\ttfamily master} locally (feel free to amend any minor problems in the author\textquotesingle{}s original commit if necessary).
\item If the change is good, and authored by another maintainer/collaborator, give them a "{}\+Ship it!"{} comment and let them handle the merge.
\end{DoxyEnumerate}\hypertarget{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_internal_a_p_i_2_internal_a_p_i_2wwwroot_2lib_2f649a80c2bb62ca06354bea1a086dc0e_autotoc_md700}{}\doxysubsection{\texorpdfstring{Submitting changes}{Submitting changes}}\label{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_internal_a_p_i_2_internal_a_p_i_2wwwroot_2lib_2f649a80c2bb62ca06354bea1a086dc0e_autotoc_md700}

\begin{DoxyEnumerate}
\item All non-\/trivial changes should be put up for review using Git\+Hub Pull Requests.
\item Your change should not be merged into {\ttfamily master} (or another feature branch), without at least one "{}\+Ship it!"{} comment from another maintainer/collaborator on the project. "{}\+Looks good to me"{} is not the same as "{}\+Ship it!"{}.
\item Try to avoid using Git\+Hub\textquotesingle{}s merge button. Locally rebase your change onto {\ttfamily master} and then push to Git\+Hub.
\item Once a feature branch has been merged into its target branch, please delete the feature branch from the remote repository.
\end{DoxyEnumerate}\hypertarget{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_internal_a_p_i_2_internal_a_p_i_2wwwroot_2lib_2f649a80c2bb62ca06354bea1a086dc0e_autotoc_md701}{}\doxysubsection{\texorpdfstring{Releasing a new version}{Releasing a new version}}\label{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_internal_a_p_i_2_internal_a_p_i_2wwwroot_2lib_2f649a80c2bb62ca06354bea1a086dc0e_autotoc_md701}

\begin{DoxyEnumerate}
\item Include all new functional changes in the CHANGELOG.
\item Use a dedicated commit to increment the version. The version needs to be added to the {\ttfamily CHANGELOG.\+md} (inc. date) and the {\ttfamily package.\+json}.
\item The commit message must be of {\ttfamily v0.\+0.\+0} format.
\item Create an annotated tag for the version\+: {\ttfamily git tag -\/m "{}v0.\+0.\+0"{} v0.\+0.\+0}.
\item Push the changes and tags to Git\+Hub\+: {\ttfamily git push -\/-\/tags origin master}.
\item Publish the new version to npm\+: {\ttfamily npm publish}. 
\end{DoxyEnumerate}
