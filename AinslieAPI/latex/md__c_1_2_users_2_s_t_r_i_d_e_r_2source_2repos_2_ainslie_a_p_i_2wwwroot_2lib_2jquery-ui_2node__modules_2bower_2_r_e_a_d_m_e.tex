Bower offers a generic, unopinionated solution to the problem of {\bfseries{front-\/end package management}}, while exposing the package dependency model via an API that can be consumed by a more opinionated build stack. There are no system wide dependencies, no dependencies are shared between different apps, and the dependency tree is flat.

Bower runs over Git, and is package-\/agnostic. A packaged component can be made up of any type of asset, and use any type of transport (e.\+g., AMD, Common\+JS, etc.).

We discourage using bower components statically for performance and security reasons (if component has an {\ttfamily upload.\+php} file that is not ignored, that can be easily exploited to do malicious stuff).

The best approach is to process components installed by bower with build tool (like \href{http://gruntjs.com/}{\texttt{ Grunt}} or \href{http://gulpjs.com/}{\texttt{ gulp}}), and serve them concatenated or using a module loader (like \href{http://requirejs.org/}{\texttt{ Require\+JS}}).\hypertarget{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_ainslie_a_p_i_2wwwroot_2lib_2jquery-ui_2node__modules_2bower_2_r_e_a_d_m_e_autotoc_md5358}{}\doxysubsection{\texorpdfstring{Uninstalling packages}{Uninstalling packages}}\label{md__c_1_2_users_2_s_t_r_i_d_e_r_2source_2repos_2_ainslie_a_p_i_2wwwroot_2lib_2jquery-ui_2node__modules_2bower_2_r_e_a_d_m_e_autotoc_md5358}
